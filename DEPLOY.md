# Деплой «Твой Голос»

## Что реально проверено, а что нет

Проверено локально (реальными командами, не на словах):
- `npm run build` (frontend) — чистая сборка
- `pip install -r requirements.txt` + импорт FastAPI-приложения — чисто
- Полный E2E цикл всех эндпоинтов через живой uvicorn + SQLite
- Нормализация `DATABASE_URL` из формата Render/Heroku в формат asyncpg

Не проверено (нет доступа к внешним сервисам из этой среды):
- Реальный деплой на Vercel/Render/Railway
- Реальное подключение к managed PostgreSQL
- Docker-сборка образа (Docker недоступен в этой среде)

Первый деплой стоит рассматривать как проверку конфигурации, а не как гарантированно рабочий процесс — типичные грабли ниже.

---

## Backend → Render

1. Создать аккаунт на render.com, подключить репозиторий с `tvoy-golos-backend`.
2. Render подхватит `render.yaml` автоматически (Blueprint), либо создать сервис вручную:
   - Runtime: Docker
   - Dockerfile: `./Dockerfile`
   - Health check path: `/api/health`
3. Создать PostgreSQL инстанс (или через тот же `render.yaml` — уже описан).
4. Переменные окружения (задать вручную, `sync: false` в render.yaml не подтягивает секреты автоматически):
   - `ANTHROPIC_API_KEY` — вставить реальный ключ
   - `FRONTEND_ORIGIN` — обновить на реальный домен после деплоя frontend
5. **Применить миграции вручную после первого деплоя** — через Render Shell:
   ```bash
   alembic upgrade head
   ```
   Это намеренно не в `CMD` Dockerfile — миграции лучше запускать контролируемо, отдельным шагом, особенно при нескольких инстансах.

### Известная ловушка, которую мы уже закрыли в коде
Render отдаёт `DATABASE_URL` в формате `postgres://...` (Heroku-совместимый), а SQLAlchemy async требует `postgresql+asyncpg://...`. Без нормализации приложение падает на старте с ошибкой диалекта. Это исправлено в `app/core/config.py` — URL автоматически переписывается независимо от формата, в котором его выдаст платформа.

### Backend → Railway (альтернатива)
Railway обычно достаточно того же `Dockerfile` — создать проект, подключить репозиторий, добавить PostgreSQL плагин (Railway сам пробросит `DATABASE_URL`, тоже в Heroku-формате — нормализация та же). Отдельный `railway.json` не создавался, так как Railway хорошо работает с Dockerfile без дополнительной конфигурации.

## Frontend → Vercel

1. Импортировать репозиторий `tvoy-golos` в Vercel.
2. Framework preset: Vite.
3. `vercel.json` уже в репозитории — настраивает SPA-роутинг (без него прямой переход по URL вроде `/diary` даст 404, потому что маршрутизация клиентская через React Router).
4. Переменная окружения:
   - `VITE_API_URL` = `https://<ваш-backend-домен>.onrender.com/api`
5. Deploy.

**Важно:** после деплоя backend и получения его финального URL — вернуться в Render/Railway и обновить `FRONTEND_ORIGIN` на реальный Vercel-домен, иначе CORS заблокирует запросы с продакшн-фронтенда (при разработке это не заметно, потому что `localhost` уже в списке допустимых origins).

## Чеклист после первого деплоя

- [ ] `curl https://<backend>/api/health` возвращает `{"status":"ok","db":true,...}`
- [ ] Открыть frontend в браузере, DevTools → Network — запросы к `/api/benefits` не блокируются CORS
- [ ] Пройти полный цикл вручную: регион → категория → льготы → генерация → consent → инструкции
- [ ] Проверить что `/docs` (Swagger) недоступен в проде — `docs_url` отключается при `ENV=production` в `main.py`

## Стоимость (ориентировочно, на момент написания)
Render Starter (web) + Starter (Postgres) и Vercel Hobby — у обоих есть бесплатные/дешёвые тарифы для MVP с низким трафиком. Точные цифры стоит сверить на сайтах платформ — они меняются.
